# 健康检查改进 - 分布式锁与错峰执行

## 问题背景

之前存在的问题是：当多个实例同时执行健康检查时，可能会触发云端API的速率限制。这不仅浪费API配额，还可能导致不必要的请求失败。

## 解决方案

我们实现了以下改进：

### 1. 分布式锁机制
- 实现了SimpleDistributedLock类来处理跨实例的锁协调
- 在健康检查前先获取分布式锁，避免多个实例同时检查同一模型
- 使用过期时间防止死锁

### 2. 错峰执行策略
- 为每个模型的健康检查引入随机抖动（jitter）
- 通过随机延迟避免多个模型在同一时刻进行健康检查
- 不同实例间的健康检查时间自然错开

### 3. 速率限制控制
- 实现全局请求限制（默认50次/分钟）
- 按提供商设置独立的速率限制
- 请求计数器定期重置

### 4. 智能健康检查
- 基于模型状态调整检查频率（稳定性高的模型检查频率降低）
- 实现了轻量级健康检查请求以减少资源消耗

## 实现文件

### AdvancedModelStatusMonitor.js
- 高级健康检查监控器，包含所有改进功能
- 分布式锁集成
- 速率限制控制
- 错峰执行机制

### SimpleDistributedLock.js
- 简单的分布式锁实现
- 在生产环境中可替换为Redis、Zookeeper等专业解决方案

## 测试验证

test_advanced_health_check.js文件验证了以下功能：
- 分布式锁的有效性
- 错峰执行的实现
- 速率限制控制
- 多实例协调

## 优势

1. **避免API限流**: 通过分布式锁和速率限制，防止多个实例同时访问API
2. **减少资源消耗**: 错峰执行减少了瞬时请求高峰
3. **提高系统稳定性**: 避免因API限流导致的服务中断
4. **灵活配置**: 支持不同提供商的个性化速率限制
5. **向后兼容**: 保持与现有API的兼容性

## 部署建议

在生产环境中，建议：
1. 使用Redis或其他专业的分布式锁服务
2. 根据具体的API提供商限额调整速率限制配置
3. 监控健康检查的频率和成功率
4. 根据实际需求调整错峰执行的时间窗口
